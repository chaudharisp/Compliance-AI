Compliance & Regulatory Policy Assistant

A RAG-powered AI assistant that answers HIPAA and GDPR compliance questions using official regulatory documents — with cited, accurate responses.

Try the Assistant See Features

How It Works

1

Ingest Documents

HIPAA and GDPR full texts are chunked and embedded into a ChromaDB vector database.

2

Ask a Question

Your question is embedded and matched against the most relevant regulatory text chunks.

3

Generate Answer

The matched context is sent to a Gemini LLM, which generates a cited, grounded answer.

4

Track & Fallback

Every request is tracked. If one model's quota is hit, the system falls back to the next automatically.

Features

RAG Pipeline

Retrieval-Augmented Generation using LlamaIndex — answers are grounded in real regulatory text, not hallucinated.

Multi-Regulation Support

Switch between HIPAA and GDPR with a dropdown. Each regulation has its own vector collection and source documents.

3-Model Fallback Chain

Gemini 2.5 Pro → Flash → Flash-Lite. If one model's free-tier quota runs out, the next takes over automatically — zero downtime.

Usage Tracking & Analytics

Self-tracked request counts, token usage, and model fallbacks — persisted to disk, viewable on the Usage Dashboard.

Conversational Context

Multi-turn chat with memory — follow-up questions understand what you previously asked, like a real conversation.

Cloud-Ready Architecture

FastAPI backend deployable to Google Cloud Run. Static frontend deployable to Azure Static Web Apps. Fully containerized.

Tech Stack

Backend Python 3.11 · FastAPI · LlamaIndex

How Usage Tracking Works

The API does not expose usage counters

The Gemini API has no endpoint to check "how many requests have I used today." It simply rejects a request with HTTP 429 when your quota is exceeded. So we track everything ourselves:

  • Every request the backend makes to Gemini is counted — requests per day, requests per minute, input tokens, output tokens.
  • Counts are saved to a JSON file at backend/storage/usage_stats.json so they survive server restarts.
  • Daily auto-reset: At midnight Pacific Time, all counters reset to zero automatically.
  • Minute-level tracking: Per-minute request counts reset every 60 seconds to respect RPM limits.
  • Exhausted models: If a model returns a 429 error, it's marked as "exhausted" for the day and skipped on future requests.

The Usage Dashboard reads from the /usage API endpoint, which returns these self-tracked counters.

What about deployment?

On Google Cloud Run, the JSON file lives inside the container's filesystem. Since Cloud Run containers can be recycled, the tracking file may reset. For a production app, you'd use a database — but for a portfolio project on the free tier, the self-tracking + daily-reset design works well. If a container restarts mid-day, the worst case is the counters reset early (you get extra quota, not less).
